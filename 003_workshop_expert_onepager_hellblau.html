Use Case A – R-Code für deskriptiven Gruppenvergleich (Follow-up-Typen)

Format: Markdown + Code-Spezifikation (Pakete, Datenstruktur, Regeln, Output)
Warum top: Pair-Programming mit KI statt „Code-Würfeln“

# Aufgabe
Erstelle R-Code für die unten beschriebene statistische Aufgabe.

## Ziel
- Korrekt ausführbarer R-Code
- Nachvollziehbar kommentiert
- Reproduzierbar (klare Pakete, klare Schritte)
- Robust gegenüber typischen Eingabefehlern (soweit sinnvoll)

## Kontext
- Zielgruppe: Statistikteam / Fachreferat
- R-Umgebung: R 4.3.x
- Erlaubte Pakete: dplyr, tidyr, ggplot2, readr
- Nicht verwenden: exotische Pakete / Internetzugriffe
- Datenquelle: CSV

## Eingabedaten / Datenstruktur
### Datenbeschreibung
- Datensatzname: ae_data
- Beobachtungseinheit: Fallmeldung
- Relevante Variablen:
  - patient_id (chr)
  - center_id (chr)
  - followup_type (factor: standard / extended)
  - completeness_score (numeric)
  - case_closure_days (numeric)
  - relevant_detail_detected (0/1)
  - report_date (Date)

### Qualitätsregeln / Annahmen

- Gruppierungslogik: followup_type

## Aufgabe im Detail
1. Daten validieren (Pflichtspalten prüfen)
2. Deskriptive Statistik nach followup_type berechnen
3. Vergleich der Gruppen:

   - completeness_score (Mittelwert/Median)
   - case_closure_days (Median)
4. Ergebnis-Tabelle erstellen
5. Optional: einfache Visualisierung (ggplot2)
6. Kurze Interpretation als Kommentar im Skript (keine medizinische Schlussfolgerung)

## Ausgabeformat

- Vollständiges Skript in einem Block

### C) Prüfpunkte / Risiken

## Regeln

# Pakete laden
library(dplyr)
library(readr)
library(ggplot2)

# 1) Daten einlesen
ae_data <- read_csv("ae_followup.csv", show_col_types = FALSE)

# 2) Pflichtspalten prüfen
required_cols <- c(
  "patient_id", "center_id", "followup_type",
  "completeness_score", "case_closure_days",
  "relevant_detail_detected", "report_date"
)
missing_cols <- setdiff(required_cols, names(ae_data))
if (length(missing_cols) > 0) {
  stop(paste("Fehlende Pflichtspalten:", paste(missing_cols, collapse = ", ")))
}

# 3) Basisvalidierung
ae_data <- ae_data %>%
  mutate(
    followup_type = as.factor(followup_type),
    relevant_detail_detected = as.numeric(relevant_detail_detected)
  )
# Rückfragepunkt: Prüfen, ob relevant_detail_detected ausschließlich 0/1 enthält.

# 4) Gruppenstatistik
summary_table <- ae_data %>%
  group_by(followup_type) %>%
  summarise(
    n_cases = n(),
    n_missing_completeness = sum(is.na(completeness_score)),
    n_missing_closure_days = sum(is.na(case_closure_days)),
    n_missing_relevant_detail = sum(is.na(relevant_detail_detected)),
    prop_relevant_detail = mean(relevant_detail_detected == 1, na.rm = TRUE),
    mean_completeness = mean(completeness_score, na.rm = TRUE),
    median_completeness = median(completeness_score, na.rm = TRUE),
    median_case_closure_days = median(case_closure_days, na.rm = TRUE),
    .groups = "drop"
  )
print(summary_table)

# 5) Optionale Visualisierung
ggplot(summary_table, aes(x = followup_type, y = prop_relevant_detail)) +
  geom_col() +
  labs(title = "Anteil klinisch relevanter Details nach Follow-up-Typ", x = "Follow-up-Typ", y = "Anteil (0-1)") +
  theme_minimal()

# 6) Interpretation (technisch)
# Deskriptive Gegenüberstellung; inferenzstatistische Aussagen erfordern weitere Tests/Modelle und fachliche Prüfung.

Kurze Erläuterung / Prüfpunkte (gekürzt)

  • CSV wird eingelesen und Pflichtspalten geprüft.
  • Fehlende Werte werden je Variable pro Gruppe ausgewiesen.
  • Ergebnis ist deskriptiv – keine inferenzstatistische Schlussfolgerung.
  • Prüfrisiken: 0/1-Codierung, Schreibvarianten in followup_type, Ausreißer, Dubletten.

# Guideline-Auszug: Prüfung von Arzneimittelbezeichnungen (Workshop-Demo)

Dieser Auszug dient als Demonstrationsmaterial für eine strukturierte Vorprüfung.

## Regeln (Beispiel)
### R1 – Irreführung vermeiden
Bezeichnungen dürfen nicht irreführend sein.

### R2 – Verwechslungsgefahr prüfen

# Begriffsdefinitionen / Glossar (Workshop-Demo)

## Abschlussschreiben (Entwurf)
Ein formaler Textentwurf zur Mitteilung eines Prüfergebnisses oder Zwischenstands.
Im Demo-System immer als Entwurf – keine finale Freigabe.

## Befund
Strukturierte Feststellung im Rahmen der Vorprüfung (z. B. „vorhanden“, „unklar“, „nicht belegt“).
Keine finale Entscheidung.

## Human Review / manuelle Fachprüfung
Verpflichtende fachliche Prüfung durch qualifiziertes Personal vor finaler Bewertung oder Freigabe.

## Verwechslungsgefahr (Demo-Begriff)
Hinweis auf mögliche Ähnlichkeit/Verwechselbarkeit mit anderen Bezeichnungen.
Belastbare Bewertung kann zusätzliche Referenzdaten erfordern.

Praxis-Hinweis: Ein Glossar stabilisiert Tonalität und Begriffsnutzung im GPT. Das reduziert uneinheitliche Formulierungen in Prüfbericht und Schreiben.
